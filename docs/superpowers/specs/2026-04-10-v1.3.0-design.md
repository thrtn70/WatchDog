# WatchDog v1.3.0 вҖ” Design Specification

**Date:** 2026-04-10
**Status:** Draft
**Author:** thrtn70

---

## Overview

v1.3.0 is a major update that transforms WatchDog from a capable game clipper into an intelligent, AI-aware recording tool. The release spans six feature areas across five build phases, with the centerpiece being a local AI-powered audio highlight detection engine that works for any game.

### Goals

- Fix the system tray context menu text color blending bug
- Add multi-layer recording status indicators (tray icon, status bar, in-game overlay)
- Replace the broken R6 Siege highlight detector with AI-powered audio classification
- Provide AI audio detection as a universal fallback for all 100+ detected games
- Build a per-game recording profiles UI with smart game-launch prompts
- Upgrade clip management with search, batch operations, tagging, and better thumbnails

### Non-Goals

- Cloud-based AI processing (stays local and private)
- Visual/screen analysis AI (deferred to future release)
- Social sharing features (Medal-style cloud links)
- New game-state integration detectors (CS2/Valorant/OW2 remain as-is)

### Release Strategy

- Multiple pushes to `main` over time, each passing 4x audit (code-reviewer + security-reviewer + typescript-reviewer + Codex CLI)
- No GitHub release created until explicitly authorized
- Version bump to 1.3.0 happens on the final push before release

## Phase 1: System Tray Overhaul

### 1.1 Fix Text Color Blending (Bug Fix)

**Problem:** The current context menu is built programmatically in `App.xaml.cs` via `H.NotifyIcon.TaskbarIcon` with no custom theme. Default Windows `ContextMenu` styling bleeds through, causing a white accent strip on the left side and poor text contrast against the dark background.

**Solution:** Convert the programmatic context menu to a XAML-defined `ContextMenu` resource with explicit dark theme styling.

**Styling:**
- Background: `--bg-base` equivalent вҶ’ `oklch(11% 0.015 195)` вҶ’ approx `#171B21`
- Text: `--text-primary` вҶ’ `oklch(88% 0.015 195)` вҶ’ approx `#D4DAE0`
- Hover background: `--bg-elevated` вҶ’ `oklch(19% 0.02 195)` вҶ’ approx `#252B33`
- Separator: `--border-subtle` вҶ’ `oklch(22% 0.015 195)` вҶ’ approx `#2D3339`
- Remove the default Windows left icon column that creates the white strip
- Apply styles via `App.Resources` or a dedicated `TrayContextMenuStyle` in the theme dictionary

**Files affected:**
- `App.xaml.cs` вҖ” remove `CreateTrayContextMenu()` programmatic builder
- New: `Resources/TrayContextMenu.xaml` or inline in `App.xaml`
- `TrayIconViewModel.cs` вҖ” bind dynamic menu item properties

### 1.2 Animated Tray Icon States

**4 icon variants** swapped based on `CaptureState`:

| State | Icon | Color Tint | Behavior |
|-------|------|-----------|----------|
| Idle | Default WatchDog icon | Neutral/gray | Static |
| Buffering | WatchDog icon | Green tint | Static (replay buffer active, ready) |
| Recording | WatchDog icon | Red tint | Subtle pulse animation |
| Saving | WatchDog icon | Amber/yellow | Static (transient, during clip save) |

**Implementation:**
- 4 pre-rendered `.ico` files in `Resources/Icons/` (already has `tray-idle.ico` and `tray-recording.ico`)
- Add: `tray-buffering.ico`, `tray-saving.ico`
- New property on `TrayIconViewModel`: `CurrentTrayIcon` driven by `CaptureState`
- Subscribe to `BufferStateChangedEvent` and `ClipSavedEvent` for state transitions
- Saving state is transient: set on clip save start, revert to previous state after 3 seconds

## Phase 3: AI-Powered Audio Highlight Detection

### 3.1 Architecture

**New namespace:** `WatchDog.Core.Highlights.Audio`

**New NuGet dependencies:**
- `Microsoft.ML.OnnxRuntime` вҖ” core inference engine
- `Microsoft.ML.OnnxRuntime.DirectML` вҖ” optional GPU acceleration (NVIDIA, AMD, Intel)

**New files:**
- `AudioHighlightDetector.cs` вҖ” implements `IHighlightDetector`
- `AudioClassifier.cs` вҖ” ONNX model wrapper
- `MelSpectrogramProcessor.cs` вҖ” audio-to-spectrogram DSP
- `AudioEventMapping.cs` вҖ” maps model output classes to `HighlightType`

### 3.2 Model

- Pre-trained **YAMNet** (Google) or **PANNs** model converted to ONNX format
- Model size: ~15-30MB
- Bundled with the app in `Resources/Models/audio-classifier.onnx`
- Classifies audio into event categories: gunshot, explosion, crowd/cheer, announcer voice, UI notification, ambient/silence

### 3.3 Detection Pipeline

```
Game Audio (OBS) вҶ’ Buffer 960ms chunks вҶ’ Mel Spectrogram (CPU DSP)
    вҶ’ ONNX Inference вҶ’ Event Classification вҶ’ Confidence Filter
    вҶ’ HighlightDetectedEvent (via IEventBus)
```

1. **Audio tap** вҖ” read from the game audio track already captured by OBS
2. **Chunk buffer** вҖ” accumulate ~960ms audio windows (YAMNet's native input size)
3. **Mel spectrogram** вҖ” lightweight DSP conversion on CPU (~1-2ms)
4. **ONNX inference** вҖ” classify the spectrogram into event categories (~5-10ms on CPU, ~1-2ms on DirectML GPU)
5. **Event mapping** вҖ” map classified audio events to `HighlightType`:
   - Gunshot cluster + kill confirmation вҶ’ `HighlightType.Kill`
   - Rapid burst (3+ in 2s) вҶ’ `HighlightType.Multikill`
   - Announcer/crowd cheer вҶ’ `HighlightType.RoundWin`
   - Game-over audio pattern вҶ’ `HighlightType.MatchEnd`
6. **Confidence filter** вҖ” only events above threshold (configurable per-game, default: 0.6) trigger highlights
7. **Publish** вҖ” `HighlightDetectedEvent` through the existing event bus with source tagged as "AI Audio"

### 3.4 Runtime Selection

- Auto-detect: try DirectML (GPU) first вҶ’ fall back to CPU
- CPU inference for audio models: ~5-10ms per 960ms chunk вҖ” well within real-time budget
- Zero measurable impact on game FPS

### 3.5 Calibration

- First 30 seconds of gameplay establish the audio baseline for that session
- Threshold auto-adjusts based on the game's audio dynamics (loud games get higher baseline)
- Per-game sensitivity slider in Game Profiles: "Less clips вҶҗ вҶ’ More clips" (maps confidence threshold 0.8 вҶ’ 0.4)

### 3.6 R6 Siege Migration

- **Remove:** `R6HighlightDetector.cs`, `R6LogLineParser.cs`, `R6GameState.cs`
- R6 Siege falls back to `AudioHighlightDetector` automatically via `HighlightDetectorRegistry`
- R6's distinctive audio cues (operator abilities, defuser beeps, announcer callouts) should produce reasonable detection at medium sensitivity

### 3.7 Universal Fallback

- `HighlightDetectorRegistry` updated: if no dedicated detector exists for a game, `AudioHighlightDetector` activates automatically
- Game-launch toast shows "Highlights (AI)" when audio detection is active vs "Highlights" for native game-state integration
- Status bar shows highlight source: "вң“ Highlights" (native) vs "вң“ Highlights (AI)" vs "вң— No highlights"

### 3.8 Known Limitations

- Less precise than game-state integration (CS2 GSI, Valorant lockfile). Will occasionally clip non-highlight moments or miss quiet kills
- Works best for games with strong audio feedback (FPS, fighting games)
- Poor fit for strategy/sim/sandbox games вҖ” game-launch toast will not offer highlights for these genres
- Model accuracy depends on the game's audio mix; players with music/voice chat at high volume may get more false positives

## Phase 4: Smart Game-Launch Recording Profiles

### 4.1 Game Profiles Settings Tab

**New tab:** "Game Profiles" in `SettingsWindow`

**Layout:**
- **Left panel** вҖ” scrollable game list, grouped:
  - "Configured" section (games with custom profiles) вҖ” at top
  - "Detected" section (games WatchDog has seen, no custom profile) вҖ” below
  - Search box at top of the list
- **Right panel** вҖ” profile editor for selected game:
  - Recording mode: radio buttons вҖ” Highlight / Session Recording / Replay Buffer
  - Resolution override (dropdown or "Use default")
  - Bitrate override (slider or "Use default")
  - Buffer duration override (replay buffer mode only)
  - Highlight sensitivity slider (AI detection only): "Less clips вҶҗ вҶ’ More clips"
  - "Reset to defaults" button

**Data model:**
Uses the existing `GameRecordingProfile` in `AppSettings`. No schema changes needed вҖ” just building the UI.

### 4.2 Game-Launch Smart Toast

**Trigger:** `GameDetectedEvent` fires from `ProcessGameDetector`

**Evaluation flow:**
1. Does this game have a saved profile? вҶ’ Apply silently, no toast
2. Is this a new game? вҶ’ Show toast with mode options
3. Is this game incompatible with highlights? вҶ’ Toast recommends session or buffer

**Toast design:** Matches existing `ClipSavedToast` style. Corner position, non-blocking, auto-dismisses after 15 seconds.

**Scenario 1 вҖ” New game, highlights supported (dedicated detector):**
```
рҹҺ® Valorant detected
Recording mode: Highlights
[Highlights] [Session] [Replay Buffer]
вҳҗ Remember for this game          [auto-dismiss 15s вҶ’ Replay Buffer]
```

**Scenario 2 вҖ” New game, AI audio highlights available:**
```
рҹҺ® Apex Legends detected
AI highlights available for this game.
[Highlights (AI)] [Session] [Replay Buffer]
вҳҗ Remember for this game          [auto-dismiss 15s вҶ’ Replay Buffer]
```

**Scenario 3 вҖ” Game incompatible with highlights (strategy/sim/racing genre):**
```
рҹҺ® Civilization VI detected
Highlights aren't reliable for this game.
Recommended: Session recording (full coverage) or
Replay buffer (saves space).
[Session] [Replay Buffer]
вҳҗ Remember for this game          [auto-dismiss 15s вҶ’ Replay Buffer]
```

**Scenario 4 вҖ” Known game with saved profile:**
No toast. Silent activation. Status bar updates.

**Incompatibility determination:**
- `GameDatabase` extended with a `GenreHint` field: FPS, MOBA, Fighting, Racing, Strategy, Sim, Sandbox, Unknown
- FPS / MOBA / Fighting вҶ’ offer highlights (dedicated or AI)
- Strategy / Sim / Racing / Sandbox вҶ’ skip highlights, recommend session or buffer
- Unknown вҶ’ offer AI highlights with note that accuracy may vary

**"Remember for this game" checkbox:**
When checked, saves the user's selection as a `GameRecordingProfile` in `AppSettings`. Next launch of that game is silent (Scenario 4).

**Auto-dismiss behavior:**
If the toast is ignored for 15 seconds, it dismisses and defaults to Replay Buffer mode. This is the safest default вҖ” captures moments without filling disk with full session recordings.

### 4.3 Profile Auto-Switching

- When `GameDetectedEvent` fires, `SessionRecordingHostedService` checks for a saved `GameRecordingProfile`
- If found: applies resolution, bitrate, buffer duration, and recording mode overrides before starting capture
- When `GameExitedEvent` fires: reverts to default settings
- If no profile and no toast interaction: applies Replay Buffer as default

## Phase 5: Clip Management Upgrades

### 5.1 Search & Filter

**New toolbar** at the top of the clip library view in `MainWindow`.

**Components:**
- **Text search** вҖ” `TextBox` with placeholder "Search clips...". Filters by clip name and game name. Debounced 300ms
- **Game filter** вҖ” `ComboBox` populated from games in the session repository
- **Date range** вҖ” quick picks: Today / This Week / This Month / Custom (date picker)
- **Highlight type filter** вҖ” multi-select chips from `HighlightType` enum: Kill, Ace, Multikill, Round Win, Death, etc.
- **Duration filter** вҖ” "Under 30s / 30sвҖ“2min / Over 2min"
- **Clear all** button

**Behavior:**
- Filters are composable (AND logic)
- Results update in real-time as filters change
- Filter state persists across app sessions in `AppSettings.ClipFilterState`
- Empty state shows "No clips match your filters" with a clear-filters link

### 5.2 Batch Operations

**Selection model:**
- Ctrl+click for individual toggle
- Shift+click for range select
- Ctrl+A selects all within current filter results

**Selection UI:**
- Selected clips get a teal border highlight and checkbox overlay
- Toolbar shows: "3 clips selected" count when selection > 0
- Batch action buttons appear in toolbar when selection active:

**Actions:**
- **Delete** вҖ” confirmation dialog: "Delete 3 clips? This cannot be undone." Removes files and metadata
- **Export** вҖ” `FolderBrowserDialog` to choose destination, copies selected clips
- **Tag** вҖ” opens a tag picker to apply/remove tags to all selected clips
- **Keyboard shortcuts** вҖ” Delete key for batch delete, Escape to clear selection

### 5.3 Clip Tagging

**Data model:**
- Add `Tags` property (`List<string>`) to `ClipMetadata`
- Stored in the clip's JSON metadata file alongside existing fields

**UI:**
- **Add tags** вҖ” "+" button on each clip card. Autocomplete dropdown from previously used tags
- **Predefined suggestions** вҖ” "montage", "funny", "clutch", "tutorial", "trash" (user can type any custom tag)
- **Tag chips** вҖ” displayed on clip cards below the thumbnail. Colored by first-letter hash for visual distinction
- **Filter by tag** вҖ” tags appear as filter chips in the search toolbar alongside highlight type filters
- **Batch tagging** вҖ” select multiple clips, click Tag in toolbar, apply/remove tags to all

### 5.4 Improved Thumbnails

**Intelligent frame selection:**
- Instead of frame 0, sample 5 evenly-spaced frames from the clip via FFmpeg
- Score each by visual complexity (pixel intensity variance)
- Select the frame with the highest score вҖ” avoids black frames, loading screens, menus
- Run via `FFmpegClipEditor` on a background thread

**Game icon overlay:**
- Small game icon (16x16) in the bottom-left corner of the thumbnail
- Extracted from the game's `.exe` via `System.Drawing.Icon.ExtractAssociatedIcon()`
- Cached in memory by game executable path

**Highlight type badge:**
- Small colored badge in the top-right corner of the thumbnail
- Color-coded: red = Kill, gold = Ace, blue = Round Win, green = Multikill, gray = Other
- Shows the highlight type text abbreviated: "K", "ACE", "RW", "MK"

**Performance:**
- Thumbnail generation runs on background threads via the existing `ThumbnailStripCache` service
- Generated thumbnails are cached to disk to avoid re-processing on app restart
